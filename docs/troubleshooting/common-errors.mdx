This guide covers common errors you might encounter when working with MCP servers and how to resolve them.

## Connection Issues

### Server fails to start or connect

- **Use absolute paths**: Specify the full path to your server
  - ✅ Good: `/Users/yourname/project/server.py`
  - ❌ Bad: `./server.py` or `server.py`

- **For `uv` projects**: Use the `--directory` flag with absolute path

  ```bash
  uv run --directory /Users/username/weather-mcp mcp.py
  ```

- **Test your command first**: Run the server command manually in a terminal to verify it works before adding it to Inspector

- **For HTTP servers**: Verify the URL is accessible
  - Test with `curl` or a browser first

## Authentication Issues

### Server fails to connect with authentication errors

**If your server is failing to connect, first verify you've selected the correct authentication option:**

- **No Authentication** - Use this for local or development servers that don't require authentication
- **Bearer Token** - Use this if you already have an API token from the server
- **OAuth 2.0** - Use this for servers that require OAuth authorization flow

### Getting 401 Unauthorized errors

If you're seeing **401 Unauthorized** errors when trying to connect to your server, this typically means the server requires authentication that hasn't been set up yet.

**Solution:**

1. Change your authentication type to **OAuth 2.0** in the server configuration
2. Go through the OAuth authorization flow to obtain credentials
3. Use the OAuth Debugger to troubleshoot any issues with the OAuth process

<Info>
  The OAuth Debugger helps you step through each part of the authorization
  process and identify where things might be going wrong.
</Info>

### Common Configuration Mistakes

- Missing required environment variables
- Incorrect transport type selection
- Typos in server URLs or commands
- Missing file permissions or executable flags

## Self-hosted Convex: WebSocket 1011 InternalServerError

If you use **self-hosted Convex** (e.g. `fletch-convex` or your own Convex backend) and the browser shows **WebSocket closed with code 1011: InternalServerError** (sometimes followed by "WebSocket reconnected at t=…"), the Convex **backend** is closing the connection. The fix is in the backend environment, not in the Inspector app.

**Cause:** Often the backend cannot validate the JWT (e.g. Cognito) because JWT-related environment variables are not set (or differ from the app).

**Fix:**

1. **Ensure the Convex backend has JWT config** – Same issuer and JWKS as the app:
   - `JWT_ISSUER` (e.g. Cognito issuer URL)
   - `JWT_JWKS_URL` (or `AWS_REGION` + `USER_POOL_ID` for Cognito)
   - `JWT_AUDIENCE` if your app validates `aud`
2. **If using `fletch-convex`** – Set these in the backend service environment (see `fletch-convex/README.md`). The `docker-compose.yml` there passes them through; add a `.env` or set them in your deployment.
3. **Set public origin** – If the app is at `https://sandbox-mcp-inspector.fletch.co` and Convex at `https://sb-convex-cloud.fletch.co`, set `CONVEX_CLOUD_ORIGIN=https://sb-convex-cloud.fletch.co` (and `CONVEX_SITE_ORIGIN` if needed) in the backend env. Unset or localhost values can cause 1011.
4. **Debug** – On the backend host, run `fletch-convex/scripts/debug-1011-capture.sh` (or set `RUST_LOG=debug`, restart, then `docker compose logs -f backend`). Reproduce the 1011 in the browser and read the backend log line at disconnect time. See `fletch-convex/README.md` for a table of common backend messages and fixes.

The Inspector client will log a short hint when it receives this disconnect (e.g. "If you're not signed in, sign in and refresh") and, for 1011, a pointer to the debug script. Convex then auto-reconnects; fixing the backend JWT/origin config reduces or eliminates 1011s.

## "This Convex deployment does not have HTTP actions enabled"

This error appears when the **Inspector server** calls Convex (e.g. for chat/stream, `/web/authorize`, tools list) but the URL it uses points at the **sync** port instead of the **HTTP-actions** port. Self-hosted Convex often has two endpoints:

- **Sync (e.g. port 3210)** – WebSocket/sync; used by the **client** via `VITE_CONVEX_URL`.
- **HTTP actions (e.g. port 3211 or site host)** – HTTP endpoints like `/stream`, `/web/authorize`; used by the **Inspector server**.

**Fix:** On the **Inspector server** (the Node process), set the environment so the effective Convex URL is the **HTTP-actions** base:

1. **Preferred:** Set `CONVEX_HTTP_URL` to the HTTP-actions base and leave `CONVEX_SELF_HOSTED_URL` unset (or set it to the same value).
   - Examples: `CONVEX_HTTP_URL=https://sb-convex-site.fletch.co` or `CONVEX_HTTP_URL=http://127.0.0.1:3211`.
2. **If you set both:** The server uses `CONVEX_SELF_HOSTED_URL` first (see `server/config.ts`). So either set both to the **same** HTTP-actions URL, or leave `CONVEX_SELF_HOSTED_URL` unset and only set `CONVEX_HTTP_URL` to the HTTP-actions base.
3. **Wrong:** Do not point the server at the sync URL only (e.g. `CONVEX_SELF_HOSTED_URL=https://sb-convex-cloud.fletch.co` with nothing on 3211). That causes "HTTP actions enabled" errors.

After changing env, restart the Inspector server and retry.

## POST /api/web/chat-v2 returns 400 Bad Request

The response body is JSON with `code` and `message`. Check the **message** to see why the request was rejected.

| Cause | `code` | `message` (or similar) |
|-------|--------|------------------------|
| Missing or invalid request body | `VALIDATION_ERROR` | "Invalid JSON body" |
| Missing/invalid hosted fields | `VALIDATION_ERROR` | Zod message, e.g. "workspaceId" / "selectedServerIds" required |
| No messages or empty array | `VALIDATION_ERROR` | "messages are required" |
| Missing or unsupported model | `VALIDATION_ERROR` | "model is not supported" |
| Anthropic model with invalid tool names | `VALIDATION_ERROR` | "Invalid tool name(s) for Anthropic..." |
| **Non–MCPJam model in hosted mode** | `FEATURE_NOT_SUPPORTED` | "Only MCPJam hosted models are supported in hosted mode" |

**Hosted mode:** At `https://sandbox-mcp-inspector.fletch.co` only **MCPJam-provided** models (e.g. from `/api/web/models`) are allowed. If the client sends a custom or third-party model id (e.g. `openai/gpt-4o` or a custom provider id), the server returns 400 with `FEATURE_NOT_SUPPORTED`. Use a model from the MCPJam models list and ensure the request body includes `workspaceId`, `selectedServerIds`, `messages`, and `model` with an MCPJam model id.

Server logs: for 400 responses the server logs `[web/chat-v2] 400 Bad Request: <message> (<code>)` so you can see the reason in ECS/CloudWatch.

## POST /api/web/servers/validate returns "Authorization failed (404)" or NOT_FOUND

The Inspector server calls **Convex** at `CONVEX_HTTP_URL` + `/web/authorize` (trailing slashes on the base are stripped). A **404** means that URL is wrong or the HTTP action is not deployed.

**Causes:**

1. **Missing `/http` path on the base URL** – Many self-hosted setups use one hostname for sync and HTTP actions with a **path prefix** for HTTP actions (see `CONVEX_SITE_ORIGIN` on the Convex backend, e.g. `https://sb-convex-cloud.fletch.co/http`). The Inspector must use that **same** base including `/http`:
   - ✅ `CONVEX_HTTP_URL=https://sb-convex-cloud.fletch.co/http` → request goes to `.../http/web/authorize`
   - ❌ `CONVEX_HTTP_URL=https://sb-convex-cloud.fletch.co` → request goes to `.../web/authorize` on the **sync** side → **404**
   If you use a **dedicated site hostname** (e.g. `https://sb-convex-site.fletch.co`) with no path prefix, set `CONVEX_HTTP_URL` to that host only.
2. **Convex backend has no `/web/authorize` HTTP action** – Deploy `convex/http.ts` from the app repo (`npx convex deploy` to the cloud URL). If cloud and site are separate tasks, use **shared modules storage** (e.g. S3) so the site serves the same routes.
3. **Wrong host** – `CONVEX_HTTP_URL` must be the **HTTP-actions** entrypoint (site host or `.../http`), not the bare sync URL when sync does not serve HTTP actions at `/web/authorize`.

**What to do:**

- Ensure the **Convex backend** (e.g. `fletch-convex` or your Convex app) defines and deploys the **HTTP action** that handles `POST /web/authorize` and that the host/port behind `CONVEX_HTTP_URL` is the one serving that action.
- Verify the Inspector server env: `CONVEX_HTTP_URL` should be the base URL of the HTTP-actions/site service (e.g. `https://sb-convex-site.fletch.co`), not the sync URL.

After the fix, the Inspector will return **404** with `code: "NOT_FOUND"` and a message pointing at the authorize endpoint and `CONVEX_HTTP_URL`. Server logs will show `[web/auth] Convex POST /web/authorize returned 404` with the URL used.
